# DataSense AI - Contributor Guidelines
## Development Standards, Git Workflows & DoD Checklists

---

## 1. Code Standards & Linting Rules

All contributors must conform to these syntax rules and styling rules before submitting code modifications.

### Python Coding Rules
*   **Style Guide:** PEP 8 compliance is enforced using the **Ruff** linter/formatter.
*   **Rule checks:** Standard lines length is capped at `120` characters. Import groups are sorted alphabetically using isort rules.
*   **Type Hinting:** Strict type hint parameters are required for all function declarations:
    ```python
    async def get_dashboard_summary(dashboard_id: UUID) -> DashboardResponse:
    ```

### TypeScript & React Guidelines
*   **Compiler Config:** TypeScript must compile cleanly in strict mode. Use explicit types, avoiding the `any` keyword.
*   **React hooks:** UI components must focus on visual layout. Move component-level state and logic to reusable React hooks.

---

## 2. Git Workflow (GitFlow)

```
main          [Stable Production Releases]
  ▲
release/      [Staging validation & release testing]
  ▲
develop       [Central integration branch for development]
  ▲
feature/*     [Local branch code changes for individual issues]
```

### Branch Naming Conventions
*   **Features:** `feature/DS-<issue_id>-<short-description>` (e.g., `feature/DS-101-jwt-tenant-claims`).
*   **Fixes:** `bugfix/DS-<issue_id>-<short-description>` (e.g., `bugfix/DS-204-duckdb-connection-leak`).
*   **Releases:** `release/v<version-number>` (e.g., `release/v1.0.0`).

### Commit Formatting Rules (Conventional Commits)
```
<type>(<scope>): <short summary description>

[Optional details body explaining rationale]
```
Allowed Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`.

---

## 3. Definition of Done (DoD)

### For a Feature
* All functions have explicit type annotations.
* Core execution logic is covered by unit tests (target: $\ge 85\%$ coverage).
* Frontend components render without compile warnings or console errors.

### For an API Endpoint
* FastAPI routes validate request payloads using Zod or Pydantic schemas.
* API responses use standard HTTP status codes, and error payloads conform to the global schema.
* All routes are documented using OpenAPI/Swagger annotations.

---

## 4. Code Review Checklist

When reviewing pull requests, verify the following:

*   [ ] **Tenant Checks:** Do database query routers validate tenant permissions?
*   [ ] **Sanitization:** Are dynamic parameters sanitized to prevent SQL injection?
*   [ ] **DuckDB Pool Management:** Are connection sessions handled correctly using context managers?
*   [ ] **Cache Invalidation:** Are cache namespaces invalidated when datasets or dashboard configurations change?
*   [ ] **Code Coverage:** Does code coverage meet the $85\%$ unit test threshold?

---

## 5. Security & AI Development Rules
1.  **PII Handling:** Column profiling checks must automatically identify and mask PII fields.
2.  **SQL Validation:** SQL queries generated by LLMs must pass AST verification checks before execution.
3.  **Secrets Management:** Never commit passwords, tokens, or private keys to git repositories. Load configuration values using environment parameters.
